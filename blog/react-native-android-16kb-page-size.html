React Native & the Android 16 KB Page Size Change: Everything You Need to Know

Google's new 16 KB memory page requirement is already live in Android 15. Here's what it means for your React Native app, your native modules, and what you must do before the deadline.

MR
Maksudur Rahman
Senior Mobile Engineer · React Native & Android
12 min read · Nov 2025

If you maintain a React Native app that ships to Android, there is one change you cannot afford to ignore in 2025. Google has changed the memory page size requirement for Android 15 from 4 KB to 16 KB — and starting November 1, 2025, all new apps and updates submitted to Google Play must be compatible.

This isn't a React Native-specific change. It's a kernel-level, hardware architecture decision by Google — but React Native developers feel it differently than pure web or Java/Kotlin developers, because React Native apps bundle native shared libraries (.so files) compiled and linked outside the standard Android build toolchain.

"The 16 KB change isn't about your JavaScript. It's about every native library your app loads — including ones you've never touched."

What is a memory page, and why does its size matter?

When an operating system loads an executable or a shared library (.so file) into RAM, it doesn't load the entire file at once. It maps the file in fixed-size chunks called memory pages. On virtually all Android devices up to this point, that page size was 4 KB (4,096 bytes).

Apple moved to 16 KB pages on Apple Silicon in 2020. Google is now following suit for Android 15 on ARM64 hardware. The reason: modern ARM64 processors can address memory in 16 KB chunks natively, which means fewer TLB (Translation Lookaside Buffer) entries are needed to map the same amount of RAM. The result is faster memory access, fewer cache misses, and better overall system performance.

Figure 1 — 4 KB vs 16 KB page mapping for a 64 KB shared library. Fewer pages = fewer TLB entries = faster memory access.

What exactly breaks — and why?

A shared library (.so file) must be aligned to the page size it was compiled for. If a library was built with 4 KB alignment and the OS tries to load it with 16 KB pages, the segment boundaries don't line up. The dynamic linker refuses to load the library, and your app crashes at startup with an error like:

Logcat — crash on 16 KB device 
CANNOT LINK EXECUTABLE "libreactnativejni.so": "libhermes.so"
  has load commands not page-size aligned.
dlopen failed: "libmymodule.so" is not page-aligned
  (expected alignment: 16384, actual: 4096)
Caused by: java.lang.UnsatisfiedLinkError: dlopen failed

This crash only happens on Android 15 devices with 16 KB page size (currently Pixel 9 series and newer ARM64 devices running Android 15). On older devices, 4 KB-aligned libraries still work fine.

Important: Starting November 1, 2025, all new app submissions and updates to Google Play must pass the 16 KB page size compatibility check — even if your target users won't have a 16 KB device for months. The check runs at submission time.

How to check your app right now

Before doing anything else, scan your current APK or AAB to find which .so files are not 16 KB aligned. Google provides a dedicated tool for this:

Terminal — check alignment with Android 16 KB tool 
# Download the check_elf_alignment tool from Android NDK
# Or use the Play Console's Pre-launch report

# Manual check with readelf (from NDK toolchain)
$ readelf -l libyourmodule.so | grep -A1 "LOAD"

# Output: SAFE (16KB aligned)
  LOAD  0x000000 ... Align: 0x4000   # 0x4000 = 16384 = 16 KB ✓

# Output: UNSAFE (only 4KB aligned)
  LOAD  0x000000 ... Align: 0x1000   # 0x1000 = 4096 = 4 KB ✗

You can also use the Android Gradle plugin's built-in lint check, which flags non-compliant libraries at build time in AGP 8.3+:

android/app/build.gradle 
android {
  lint {
    enable "NonCompatibleWith16KbPageSizes"
    error  "NonCompatibleWith16KbPageSizes"  // fail build if found
  }
}

The fix — for your own native modules

If you write custom native modules in C/C++ using the NDK, the fix requires updating your CMake or ndk-build configuration to set the correct alignment flags:

CMakeLists.txt 
# Add these linker flags to your CMakeLists.txt
target_link_options(mymodule PRIVATE
  "-Wl,-z,max-page-size=16384"   # set max page size to 16 KB
  "-Wl,-z,common-page-size=16384" # set common page size to 16 KB
)

# Alternatively, for ALL targets in your project:
add_link_options(
  "-Wl,-z,max-page-size=16384"
  "-Wl,-z,common-page-size=16384"
)

If you use ndk-build instead of CMake:

Android.mk 
# In your Android.mk
LOCAL_LDFLAGS += -Wl,-z,max-page-size=16384
LOCAL_LDFLAGS += -Wl,-z,common-page-size=16384
NDK version matters. You need NDK r26b or later to reliably produce 16 KB-aligned shared libraries. Check your NDK version in android/local.properties or in Android Studio under SDK Manager → NDK.

What about third-party native modules?

This is where most React Native developers get stuck. Your own code is easy to fix. But your app likely pulls in 10–30 npm packages that include native code — React Native Camera, react-native-maps, react-native-vision-camera, SQLite plugins, crypto libraries, and so on. Each of them ships precompiled .so files.

Here's how to approach each dependency: if a library is already 16 KB compatible, no action is needed. If the maintainer has released a fix, update the package. Otherwise, fork and patch or replace the library.

Testing on a 16 KB emulator

You don't need a Pixel 9 to test. Android Studio 2023.3.1 Canary and later ship a 16 KB emulator image. Here's how to set it up:

Android Studio — AVD Manager 
# In AVD Manager, create a new device:
# Device: Pixel 8 Pro (or any ARM64 device)
# System Image: Android 15 (API 35), ARM64 — "16K Page Size" variant
# This image emulates the 16 KB page size environment

# Or via command line with avdmanager:
$ avdmanager create avd \
    --name "16kb_test" \
    --package "system-images;android-35;google_apis;arm64-v8a_16k" \
    --device "pixel_8_pro"
Quick win. Install your APK on the 16 KB emulator before doing anything else. If it launches without crashing, you may already be compliant. If it crashes on startup with a dlopen failed error, you now know exactly which .so file to fix first.

The full checklist — step by step

Here's the complete action plan in order of priority:

Action checklist 
## Step 1 — Upgrade React Native
target: RN 0.76+  (includes updated Hermes + NDK config)
npx react-native upgrade

## Step 2 — Update NDK in local.properties
ndk.dir=/path/to/ndk/26.3+  (r26b or later required)

## Step 3 — Enable the lint check
# In android/app/build.gradle:

## Step 4 — Scan your .so files
readelf -l android/app/build/intermediates/**/*.so \
  | grep "Align"
# All must show 0x4000 (16384) not 0x1000 (4096)

## Step 5 — Update all native dependencies
npx npm-check-updates -u

"-Wl,-z,max-page-size=16384"

## Step 8 — Submit to Play Console
# Play Console now runs 16KB compatibility check automatically

Performance impact — is it actually faster?

Yes, and the gains are measurable. On Pixel 9 hardware (16 KB native), Google reports:

Metric4 KB pages16 KB pagesDelta
App cold start time~920 ms~810 ms~12% faster
TLB miss rate (library load)baseline–~75% fewer
Library load I/O operationsN pagesN÷4 pages4× fewer ops
Binary size on disksmallerslightly larger+2–5% APK size

The APK size increase is real but minimal — ELF segment padding rounds up to the next 16 KB boundary instead of 4 KB, so the binary alignment sections are slightly larger. In practice this adds a few hundred KB to most React Native apps.

Common mistakes to avoid

Mistake 1 — Assuming upgrading RN is enough. React Native core is compatible, but every third-party library with native code must be individually verified. One non-compliant .so file crashes the entire app.
Mistake 2 — Ignoring the deadline because your current users don't have Pixel 9. Google Play enforces the compatibility check at submission time. If your AAB fails the check, the update is rejected — regardless of your target market.
Mistake 3 — Using an old NDK. NDK versions before r26b had linker bugs that could produce incorrectly aligned binaries even with the correct flags. Always use the latest stable NDK.

Conclusion

The 16 KB page size change is one of the most impactful low-level Android changes in years — but the path forward for React Native developers is clear. Upgrade to RN 0.76+, update your NDK, scan your dependencies, and test on a 16 KB emulator before the November 1, 2025 deadline.

The good news: for most apps using popular, actively maintained libraries, the bulk of the work is already done by the library maintainers. Your job is to upgrade, verify, and ship. The apps that get caught are the ones with old pinned dependency versions, unmaintained native modules, or custom NDK code that hasn't been touched in years.

Start the audit now. Don't leave it for October.
